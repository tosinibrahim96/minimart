# How to Version Your Database Schema with Alembic

> **Status: IN PROGRESS — all four acceptance criteria demonstrated (from-empty build,
> add-column with no data loss, downgrade, migrations in git); PENDING: final commits +
> spec tick via /done.**

> Put the database schema under version control: every change becomes a reviewed, reversible
> migration file, and `create_all` is retired forever.

**Phase:** 3 — Database migrations (Alembic)
**Concept it taught:** Schema versioning — migrations as a chain of diffs (git for your
database), autogenerate as a *draft* you review rather than trust, deterministic constraint
naming, and how to bring an already-existing database under migration control.
**Why it matters:** "How do you handle schema changes in production?" is a near-guaranteed
interview question. "I never edit tables by hand — every change is an autogenerated,
hand-reviewed Alembic migration, committed to git, with a working downgrade" is the senior
answer. Bonus depth: knowing *what autogenerate can't see* and why Postgres's transactional
DDL makes failed migrations safe.

## The mental model (read first): git, but for your schema

| git | Alembic |
|---|---|
| commit (a diff) | migration file (`upgrade()` / `downgrade()`) |
| commit history | `alembic/versions/` — each file's `down_revision` points at its parent |
| HEAD | `head` — newest migration in the chain |
| "which commit am I on?" | `alembic_version` table **in the database** (one row: last applied revision) |
| `git log` | `alembic history` |
| applying commits | `alembic upgrade head` |
| going back | `alembic downgrade -1` |

The extra trick git doesn't have: `alembic revision --autogenerate` connects to the live DB,
diffs it against `Base.metadata` (what the models *say* should exist), and writes the diff as
a migration — a `git diff` that writes its own commit. Which is exactly why you **read it
before applying**.

Why `create_all` had to die (`main.py`'s lifespan used it until this phase): it only creates
tables that *don't exist* — it never alters an existing one. Add a column to a model and
restart: nothing happens; model and database now silently disagree. Migrations are the
mechanism for *evolving* a schema that already has data.

## Prerequisites

- The Docker-first stack (Phase 0) with Postgres running.
- Phases 1–2: the `categories` and `products` models this phase puts under version control.
- Dependency round-trip: `../guides/dependency-management.md`.

## Steps

### 1) Install Alembic — as a MAIN dependency (a real decision)

```bash
docker compose run --rm api uv add alembic   # edits pyproject.toml + uv.lock via the mount
docker compose up -d --build                 # rebuild — bakes it into the image's /opt/venv
```

**Why main and not the dev group:** ask "*where does this command run in production?*"
pytest runs only in dev/CI → dev group. But `alembic upgrade head` runs **against the
production database, from the production image** (a release step just before new code
starts). The prod image installs with `--no-dev`, so alembic in the dev group would be
absent exactly where migrations matter most.

**Gotcha we hit:** after `uv add` via the throwaway container, running
`docker compose exec api alembic --version` failed with
`OCI runtime exec failed: exec: "alembic": executable file not found in $PATH`. The lockfile
edits survived (bind mount) but the *install* died with the `--rm` container — the running
image predated alembic. The rebuild is what makes it real. (Full model — one lockfile, two
derived environments: see the dependency-management guide.)

### 2) Scaffold: `alembic init`

```bash
docker compose exec api alembic init alembic
```

Creates:

```
alembic.ini          ← config, repo root — read at every alembic command
alembic/
├── env.py           ← Python that RUNS on every alembic command: connects + points at models
├── script.py.mako   ← template new migrations are stamped from
└── versions/        ← the migration chain (empty so far)
```

### 3) `alembic.ini` — three edits

**(a) Never hardcode the URL.** The template ships
`sqlalchemy.url = driver://user:pass@localhost/dbname` and blogs happily paste real
credentials into it — violating "all config from env, no secrets in code, ever." Comment it
out / leave it blank; `env.py` injects the real value from settings (step 4).

**(b) Human-sortable filenames.** Uncomment the `file_template` line Alembic itself ships:

```ini
file_template = %%(year)d_%%(month).2d_%%(day).2d_%%(hour).2d%%(minute).2d-%%(rev)s_%%(slug)s
```

Default names are `<hash>_<slug>.py` → `ls versions/` is meaningless hash order.
Load-bearing fact: **Alembic never orders by filename** — ordering is the `down_revision`
linked list inside each file; names are purely for humans. fastapi-best-practices recommends
date-only (`%%(year)d-%%(month).2d-%%(day).2d_%%(slug)s`), but that template has two flaws:
two same-day migrations sort alphabetically (lying about order — hour+minute fixes it), and
dropping `%%(rev)s` breaks the glance-link between `alembic current` output and a file.
Benchmark the leaders, but think.

**(c) Post-write hook: lint migrations at generation time.** Autogenerated files come with
lint noise (see Troubleshooting). Uncomment the shipped ruff block so every new migration is
auto-fixed the moment it's written:

```ini
[post_write_hooks]
hooks = ruff
ruff.type = module
ruff.module = ruff
ruff.options = check --fix REVISION_SCRIPT_FILENAME
```

Same philosophy as naming conventions: make the tool *produce* correct output instead of
correcting it by hand forever.

### 4) Wire `env.py` — the file blogs mangle most

`env.py` is code, not config — it runs in a **fresh Python process** on every alembic
command. Three edits:

```python
from app.core.config import settings
from app.core.database import Base

from app.categories import models as categories_models  # noqa: F401
from app.products import models as products_models  # noqa: F401

config = context.config
config.set_main_option("sqlalchemy.url", settings.database_url)  # from env, same as the app

target_metadata = Base.metadata   # what autogenerate diffs the live DB against
```

**Why the URL works with no value in any file:** pydantic-settings resolves
`database_url` from the `DATABASE_URL` env var (field name, case-insensitive), which
`docker-compose.yml` injects into the api container. No default declared = required =
`ValidationError` at startup if missing — fail-fast config. This is also why alembic runs
*inside the container*: the host has neither the env var nor DNS for the `db` hostname.

**Why the explicit model imports (the trap):** `Base.metadata` starts EMPTY. A model
registers into it as a side effect of its `class Product(Base):` statement executing — i.e.
when its *module is imported*. The app got this transitively (main → router → service →
models), but `env.py` imports no routers. Skip these imports and `target_metadata` is an
empty phone book — autogenerate concludes "models say no tables should exist" and generates
`op.drop_table(...)` for everything. A silently catastrophic migration, caught only by
reading. (Alternative considered: a single aggregator module importing every models module;
deferred as ceremony at two domains.)

### 5) Naming conventions — deterministic constraint names (do this BEFORE the first migration)

First autogenerate attempt produced `sa.UniqueConstraint('name')` — anonymous. Postgres
invents names for unnamed constraints (`categories_name_key`), which works until a future
migration must DROP one and you're guessing what a particular database decided to call it.
Fix: a naming convention on the metadata, in `app/core/database.py`:

```python
convention = {
    "ix": "ix_%(column_0_label)s",
    "uq": "uq_%(table_name)s_%(column_0_name)s",
    "ck": "ck_%(table_name)s_%(constraint_name)s",
    "fk": "fk_%(table_name)s_%(column_0_name)s_%(referred_table_name)s",
    "pk": "pk_%(table_name)s",
}

class Base(DeclarativeBase):
    metadata = MetaData(naming_convention=convention)
```

Key by key: `ix` index (`column_0_label` = `<table>_<column>`, table baked in) · `uq` unique
(first column only — deterministic beats descriptive) · `ck` CHECK constraint — a boolean
rule the DB itself enforces (`CHECK (price >= 0)`), the last line of defense behind
Pydantic's 422s; uses `%(constraint_name)s` (you name each check: `name="price_nonnegative"`
→ `ck_products_price_nonnegative`) because the `column_0_name` variant must *parse the SQL*
to infer a column and fails on multi-column checks · `fk` reads as a sentence:
`fk_products_category_id_categories` · `pk` one per table.

This is SQLAlchemy's documented convention and (modulo `ck`) fastapi-best-practices'. In
generated migrations, `op.f('pk_products')` means "name already final — don't re-apply
conventions."

### 6) Retire `create_all`, then solve the chicken-and-egg

Delete `Base.metadata.create_all(bind=engine)` from the lifespan (keep `engine.dispose()` —
still real cleanup). **Sequencing trap:** do this *before* nuking the DB, or the app will
helpfully rebuild the tables and sabotage the diff.

The chicken-and-egg: autogenerate = diff(live DB, metadata). Our DB already matched the
models (`create_all` built it) → diff is empty → no initial migration possible. Two ways out:

- **Path A (dev — data disposable, what we did):** make "empty" true again — nuke the volume,
  autogenerate against a blank database → full create-everything migration. Also proves the
  from-empty acceptance criterion.
- **Path B (production — data precious):** generate the initial migration against a scratch
  DB (or write it), then `alembic stamp head` on the real DB — writes the revision id into
  `alembic_version` *without running anything*: "trust me, this schema already matches." This
  is how legacy databases come under migration control. Interview gold.

```bash
docker compose down -v        # -v deletes the named volume = all Postgres data
docker compose up -d          # no --build: code is bind-mounted; only deps need rebuilds
docker compose exec api alembic revision --autogenerate -m "create categories and products tables"
```

### 7) READ the migration (the acceptance criterion is "autogenerated then reviewed")

What the review of ours actually found:

- `down_revision = None` — the chain's root, like a first commit.
- `price` survived as `sa.Numeric(precision=10, scale=2)` — the line autogenerate most often
  betrays (check it every time money is involved).
- `downgrade()` drops `products` **before** `categories` — FK dependency order, not
  alphabetical. `upgrade()` mirrors it.
- **What autogenerate can't see:** `updated_at`'s `onupdate=func.now()` is absent — it's
  Python-side behavior, not DDL; the DB has no idea (and a raw `psql` UPDATE bypasses it).
  Also invisible: column *renames* (it emits drop+add — data loss if trusted blindly) and
  anything trigger-shaped.

### 8) Apply and verify like you don't trust it

```bash
docker compose exec api alembic upgrade head
docker compose exec api alembic current                                  # → 6864754d65f9 (head)
docker compose exec db psql -U postgres -d minimart -c '\d products'     # constraint names live
# then create a category + product through the API — app runs on a schema built purely by Alembic
```

`\d products` showed `"pk_products" PRIMARY KEY` and
`"fk_products_category_id_categories" FOREIGN KEY` — the convention in the wild — plus
details worth reading consciously: `nextval('products_id_seq')` (the sequence behind the
integer PK), `timestamp with time zone`, and `now()`/`true` server defaults in real DDL.

**Decode alembic's startup lines:** `Will assume transactional DDL` = Postgres runs DDL
inside transactions, so a migration that fails halfway **rolls back completely** — no
half-altered schema. MySQL/Oracle auto-commit DDL and leave you stranded mid-change. An
underrated reason Postgres is the serious default.

### 9) The add-a-column exercise: `brand` on a table with 43 rows in it

Criterion 2 ("new column, no data loss") demands a *populated* table — so seed data first
and capture the before state (`SELECT id, name, price FROM products ORDER BY id LIMIT 5;`
plus a count). Then:

```python
# app/products/models.py
brand: Mapped[str | None] = mapped_column(String(100), nullable=True)
```

```bash
docker compose exec api alembic revision --autogenerate -m "add brand column to products table"
# read it: two lines of substance — add_column(nullable=True) / drop_column
docker compose exec api alembic upgrade head
```

After: same rows, same values, `count(brand) = 0` — 43 rows survived, brand honestly NULL.

**Why nullable — the decision, not the default:** play migration-runner: if the column were
NOT NULL, what value would the 43 *existing* rows get at the instant `ALTER TABLE` runs?
None exists → Postgres must reject it (or you need default/backfill staging — Phase 5's
`sku` lesson). Nullable is also the honest model: a brand-less product is valid.
(Considered and rejected for this exercise: `image_url` — promoted to its own Phase 13b,
object storage + presigned URLs; and a `brands` *table* + FK — brand smells like an entity,
like category, but normalizing it is three migrations + a backfill story for later.
Right-sizing to today's requirement is the senior habit.)

**Why the column choice cascaded into schema decisions** (caught in review, worth keeping):

- **Input constraints must mirror the column cap.** `brand` initially had no `max_length`:
  a 150-char brand would pass Pydantic, then die in Postgres (`value too long`) →
  unhandled `DataError` → 500. The Phase 2 contract is "bad input dies at the edge with a
  422" — every input field mirrors its column (`name` 50, `description` 255, `brand` 100),
  plus `min_length=1` so `""` can't masquerade as "no brand."
- **Output schemas describe reality; they don't police it.** A `max_length` on `ProductRead`
  *below* the column cap is a landmine: any legitimately-stored longer value would fail
  serialization of your own response. Constraints belong on the way in.
- **PATCH fields have THREE states, not two:** omitted (default applied, validators don't
  run — Pydantic skips defaults; service uses `exclude_unset`) · explicit `null` (validator
  fires) · a value. Declaring `brand: str = Field(...)` in the update schema to mean "can't
  null it" actually meant "every PATCH must include brand" — breaking partial updates. The
  correct spelling of *settable-but-not-clearable* is `str | None = Field(None, ...)` PLUS
  membership in the `reject_explicit_null` validator. (Ratchet semantics — a product can
  gain a brand but never lose one — is a deliberate product decision; be able to defend it.)
- **Product rule made explicit:** `ProductCreate.brand` is *required* — new products declare
  a brand; the 43 old rows are grandfathered as NULL. API-required + DB-nullable is
  coherent *because* of existing rows; the follow-up ("when does the column become NOT
  NULL?") is answered in stages: backfill, then constraint — Phase 5 does it for real.
- Filter guard: `min_length=1` on the brand filter, else `?brand=` (empty string) builds
  `ILIKE '%%'` — matches every branded product while silently excluding NULL brands: a
  filter that looks like no-filter but isn't.

### 10) The downgrade proof — and the schema/code mismatch window

Predict before running: with `brand` dropped but the app code still referencing it, what
does `GET /products` do?

```bash
docker compose exec api alembic downgrade -1     # drops brand
docker compose exec api alembic current          # back to the initial revision
# \d products → column gone; SELECT count(*) → still 43 (other columns untouched)
curl -s localhost:8000/products                  # ← the prediction, live
docker compose exec api alembic upgrade head     # heal; API works again
```

What happens, precisely: the ORM builds queries from the *model*, which still declares
`brand` → `SELECT ... products.brand ...` → Postgres `UndefinedColumn` → psycopg
`ProgrammingError` → nothing catches it → the client sees a raw **500**. From outside, a
schema/code mismatch doesn't announce itself — the API "just starts failing"; the truth is
only in the logs. This window is exactly the *backward-compatibility* question: in every
real deploy, old-or-new code runs against new-or-old schema for a moment.

Two more lessons this run made concrete:

- **Downgrades are not magically lossless.** `drop_column` destroys whatever the column
  holds. Ours was all-NULL, so this rollback was clean — but had brands been written first,
  `downgrade` would silently delete them. Production rollbacks are a bigger deal than the
  one-word command makes them feel.
- **An applied migration is immutable history.** Renaming/regenerating a migration file was
  fine *only* while unapplied and uncommitted (we did it to pick up the `file_template`) —
  once applied anywhere or pushed, treat it like a pushed git commit.

## Run & verify

- `docker compose exec api alembic current` → prints the head revision.
- `\d products` in psql → constraints carry convention names (`pk_`, `fk_`, `uq_`); `brand`
  is `character varying(100)`, nullable.
- API create/list works on a DB built from migrations alone, from empty. ✅ criterion 1.
- 43 pre-existing rows survived the `brand` migration; `count(brand) = 0`. ✅ criterion 2.
- `downgrade -1` removed the column (other data intact), `upgrade head` restored it; the
  broken-API window in between was observed and understood. ✅ criterion 3.
- Migration files committed. ✅ criterion 4.

## Troubleshooting (real issues we hit)

- **`exec: "alembic": executable file not found in $PATH`** → added the dep but didn't
  rebuild; the install died with the throwaway container. `docker compose up -d --build`.
- **Migration file has no date prefix despite `file_template`** → the ini is read at
  *generation* time; the file was generated before the edit. Fix (only because it was
  unapplied + uncommitted): delete the file, regenerate. An applied/pushed migration is
  immutable history — like a pushed git commit, you don't rename or edit it.
- **Autogenerate produces an EMPTY migration** → the live DB already matches the metadata
  (diff of identical things). For an initial migration: empty the DB first (Path A) or
  generate elsewhere + `stamp head` (Path B).
- **Autogenerate wants to DROP your tables** → `target_metadata` is empty because `env.py`
  doesn't import the model modules. Registration is an import side effect.
- **`W291 Trailing whitespace` in the generated migration, and `--fix` won't fix it** → the
  Alembic template emits `Revises: ` (trailing space) in the module *docstring*; whitespace
  inside a string literal changes its value, so ruff classifies the fix as unsafe. Options:
  `ruff check alembic/versions --fix --unsafe-fixes`, fix by hand, or add `--unsafe-fixes`
  to the post-write hook / per-file-ignore `W291` for `alembic/versions/*`.

## Interview talking point

"Every schema change in my project is an autogenerated, hand-reviewed Alembic migration —
reviewed because autogenerate can't see renames (it emits drop+add) or Python-side behavior
like `onupdate`. I set a naming convention on the metadata before the first migration so
every constraint has a deterministic name a future migration can DROP, and I know two ways
to bring an existing database under Alembic: rebuild from empty in dev, `alembic stamp head`
in production. And I run migrations on Postgres knowing DDL is transactional — a failed
migration rolls back clean instead of leaving a half-altered schema."
